Lab topology (Windows + Linux)

High-level components used during the module and my extensions:

  • Splunk Core: Indexer + Search Head (single instance for the lab, but workflows mimic distributed setups).
  • Ingestion: Universal Forwarder on Windows/Linux; occasional HEC tests for JSON sources.

Advanced SPL — representative patterns

A few patterns I leaned on repeatedly during investigations:

# 1) Spike detection (failed auths per user per 5 min)
index=sec auth_status=failed
| bin _time span=5m
| stats count AS fails, values(src) AS src_ips by _time, user
| where fails >= 5
| sort -_time -fails

# 2) Sessionization without transaction (streamstats)
index=web sourcetype=web:access status>=500
| rex field=uri_path "/(?<endpoint>[^?]+)"
| streamstats current=f window=1 reset_on_change=t by clientip
    earliest(_time) AS prev_time, last(endpoint) AS prev_ep
| eval gap = _time - prev_time
| where gap <= 60
| stats count AS bursts, values(endpoint) AS endpoints by clientip
| where bursts >= 3

# 3) Outlier IPs per user compared to typical geo
index=sec sourcetype=linux:sshd action=success
| iplocation src
| stats dc(Country) AS geo_diversity, values(Country) AS geos by user
| where geo_diversity > 1

# 4) JSON parsing (HEC or app logs)
index=app sourcetype=app:json
| spath
| stats count BY level, endpoint, user

# 5) Multi-stage correlation (failed logins followed by success)
index=sec (action=failed OR action=success) user=*
| stats latest(eval(if(action="success",_time,null()))) AS t_success,
        latest(eval(if(action="failed",_time,null())))  AS t_failed
  by user, src
| where t_success >= t_failed AND (t_success - t_failed) <= 600
| eval comment="Rapid success after failures — potential brute-force then valid creds"

These patterns compress noisy logs into analytical views (spikes, sessions, outliers, and sequences). They’re easy to promote into dashboard panels and alerts.

Dashboards & scheduled reports

I translated recurring questions into panels—“Where are failures spiking?”, “Which endpoints return 5xx most?”, “Which users changed geo?”. Then I backed these with saved searches and scheduled reports to maintain rhythm.

  • Auth Health: failure spikes by user/IP; success-after-fail series.
  • Web/API Reliability: 5xx by endpoint over time; outlier client IPs.
  • Geo & Velocity: country distribution by user; basic “impossible travel” approximations.
# Panel: 5xx by endpoint (1m)
index=web sourcetype=web:access status>=500
| rex field=uri_path "/(?<endpoint>[^?]+)"
| timechart span=1m count BY endpoint limit=10
Packaging tip: keep searches parameterized (e.g., $endpoint$, $window$) to reuse panels across services and shorten iteration time.

Data manipulation — props.conf / transforms.conf

Real logs are messy. I used custom stanzas to fix timestamps, break multiline events, and extract fields. A few distilled examples (paths vary by deployment):

props.conf

# Linux SSHD (example)
[linux:sshd]
TIME_PREFIX = ^\w{3}\s+\d+\s+\d\d:\d\d:\d\d\s
MAX_TIMESTAMP_LOOKAHEAD = 32
LINE_BREAKER = ([\r\n]+)
SHOULD_LINEMERGE = false
TRUNCATE = 20000
REPORT-sshd-fields = sshd-fields

# Web access (custom path extraction)
[web:access]
TIME_FORMAT = %d/%b/%Y:%H:%M:%S %z
TIME_PREFIX = \[
SHOULD_LINEMERGE = false
REPORT-web-fields = web-fields, web-useragent

transforms.conf

[sshd-fields]
REGEX = ^(?<host>\S+)\s+sshd\[\d+\]:\s(?<action>Failed|Accepted).*?from\s(?<src>\S+)\s.*?for\s(?<user>\S+)
FORMAT = src::$1 user::$2 action::$3
WRITE_META = false

[web-fields]
REGEX = ^(?<clientip>\S+)\s\S+\s\S+\s\[(?<time>[^\]]+)\]\s"(?<method>\S+)\s(?<uri_path>\S+).*"\s(?<status>\d{3})\s(?<bytes>\d+)
FORMAT = clientip::$1 method::$2 uri_path::$3 status::$4 bytes::$5

[web-useragent]
REGEX = "([^"]+)"\s\d+\s\d+\s"([^"]*)"\s"(?<useragent>[^"]+)"$
FORMAT = useragent::$1

Once fields are clean, SPL becomes dramatically simpler, and dashboards stabilize.

Capstone — retrieve & analyze a full report

I treated the final exercise as a mini-IR: pull a multi-section report (auth, web, and system events), normalize, and produce a short, defensible narrative.

  1. Scope & Data: Defined target users, time window, and sources (Linux SSHD, Windows Security, web access logs).
  2. Normalization: Ensured fields like user, src, endpoint, status, action were consistently extracted.
  3. Correlation: Linked failure spikes to eventual successes, mapped web 5xx spikes to deployment notes, and paired SSH successes to new geo.
  4. Findings: Flagged rapid success-after-fail for two accounts, identified a noisy endpoint causing 5xx bursts, and noted cross-geo login for one user.
  5. Output: A 2-page summary with indicators (IPs, users, endpoints), timelines, and recommended mitigations.
# Quick stitched view: failed→success within 10 min + 5xx adjacency
(index=sec sourcetype=linux:sshd (action=failed OR action=success))
OR (index=web sourcetype=web:access status>=500)
| eval kind=case(sourcetype="linux:sshd","auth", sourcetype="web:access","web", true(),"other")
| transaction user maxpause=600 keepevicted=t startswith=eval(kind="auth" AND action="failed") endswith=eval(kind="auth" AND action="success")
| eventstats count AS web5xx by user
| table _time user src kind action uri_path status web5xx duration
IR habit: after searches stabilize, capture them as saved searches and export a PDF/HTML snapshot alongside raw indicators. Future me will thank present me.

Challenges & resolutions

  • Timestamp drift / parsing: fixed with explicit TIME_PREFIX and TIME_FORMAT to avoid mis-bucketed events.
  • Multiline events: disabled merge for line-oriented sources; added dedicated LINE_BREAKER where needed.
  • Noisy sourcetypes: split access/error into distinct sourcetypes; kept props/transforms readable.
  • Windows vs Linux parity: favored common field names so a single SPL works across OS families.
