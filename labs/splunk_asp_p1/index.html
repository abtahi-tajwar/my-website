Splunk × ASP.NET Core (Part 1)

Writing logs with Serilog in a containerized ASP.NET Core app and forwarding them to a remote Splunk instance via Universal Forwarder.

SEIM Splunk Docker Serilog Splunk Universal Forwarder .NET 9

Overview

This experiment explores how application-level logs flow from a containerized ASP.NET Core service into a remote Splunk instance. In Part 1, the focus is on generating application logs with Serilog and shipping them using a containerized Splunk Universal Forwarder (UF) to an indexer running on another laptop.

Architecture (Part 1)

Minimal viable pipeline to validate ingestion.

[ ASP.NET Core (Docker) ] --writes--> [/app/Logs/app-YYYY-MM-DD.log]
            │
            └─(shared Docker volume: app-logs)
                │
[ Splunk Universal Forwarder (Docker, amd64) ] --TCP 9997--> [ Splunk Indexer (remote laptop) ]
single-writer file sink shared volume HEC not used (UF to 9997) custom sourcetype: app:fleetingoffers

Implementation Notes

Serilog (file sink)

// Packages
// dotnet add package Serilog.Sinks.File
// dotnet add package Serilog.Extensions.Logging

Directory.CreateDirectory("/app/Logs");
Log.Logger = new LoggerConfiguration()
  .MinimumLevel.Information()
  .WriteTo.File(
    "/app/Logs/app-.log",
    rollingInterval: RollingInterval.Day,
    retainedFileCountLimit: 7,
    fileSizeLimitBytes: 10_000_000,
    rollOnFileSizeLimit: true,
    shared: true)
  .CreateLogger();

builder.Logging.ClearProviders();
builder.Logging.AddSerilog(Log.Logger, dispose: true);

Docker Compose (core bits)

services:
  app:
    build: .
    container_name: fleetingoffers_api
    volumes:
      - app-logs:/app/Logs

  splunk-uf:
    image: splunk/universalforwarder:latest
    platform: linux/amd64          # host is arm64; UF image pinned to amd64
    container_name: splunk-uf
    environment:
      - SPLUNK_GENERAL_TERMS=--accept-sgt-current-at-splunk-com
      - SPLUNK_START_ARGS=--accept-license
      - SPLUNK_PASSWORD=${Splunk_Password}
      - SPLUNK_CMD=add forward-server ${Splunk_Host}:9997 -auth ${Splunk_Username}:${Splunk_Password}
      - SPLUNK_ADD=monitor /mnt/logs -index ${Splunk_Index} -sourcetype app:fleetingoffers
    volumes:
      - uf-etc:/opt/splunkforwarder/etc
      - uf-var:/opt/splunkforwarder/var
      - app-logs:/mnt/logs:ro

volumes:
  uf-etc:
  uf-var:
  app-logs:

Troubleshooting Highlights

  • UF exits with code 1 (license): add SPLUNK_GENERAL_TERMS and SPLUNK_START_ARGS.
  • "no matching manifest for linux/arm64": set platform: linux/amd64 for UF.
  • Permission errors via CLI: exec as -u splunk when running UF commands.
  • Forwarder not active: on indexer, enable a receiving port 9997 and verify list forward-server.

Validation & Results

  • UF shows Active forwards → ${Splunk_Host}:9997
  • Monitors includes /mnt/logs mapped from the app’s /app/Logs.
  • Events arrive in Splunk index ${Splunk_Index} with sourcetype app:fleetingoffers.

Proof-of-life complete: end-to-end ingestion working across machines.

Lessons Learned

  • Container platform mismatches are subtle—pinning UF to linux/amd64 is an easy fix on Apple Silicon.
  • Persist UF state (uf-etc, uf-var) so it doesn’t re-onboard every restart.
  • Keep file sink output parseable (consider compact JSON) to power downstream analysis.
  • Data protection warnings in ASP.NET containers are normal unless you persist keys.
